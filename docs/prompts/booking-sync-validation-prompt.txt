SYNC BEHAVIOR REFERENCE (source of truth for this audit)
================================================================================

The following is the verified Calendly ↔ Google Calendar sync model.
Our booking module must match or exceed each behavior.

BEHAVIOR 1 — Booking created → Google Calendar event created
  Calendly: New booking writes an event to the designated target calendar.
  Expected: Our BookingService.createBooking() does this. ✓ (verify only)

BEHAVIOR 2 — Google Calendar event deleted or declined BY THE COACH
           → Booking cancelled in our system
  Calendly: If the coach deletes or declines the Calendly event in GCal,
            it is also cancelled in Calendly.
  Gap risk: Our webhook handler flushes cache only. It does NOT detect
            deletions or RSVP declines and does NOT cancel the Booking
            document or notify the client.

BEHAVIOR 3 — Google Calendar event rescheduled BY THE COACH
           → Booking updated in our system (new start/end time)
  Calendly: If the coach moves a booking to a different time in GCal,
            the change syncs back into Calendly (bi-directional time sync).
  Gap risk: Our webhook handler flushes cache only. It does NOT detect
            time changes and does NOT update Booking.startTime / endTime
            or re-send a reschedule confirmation to the client.

BEHAVIOR 4 — Google Calendar event deleted BY THE CLIENT (invitee)
           → No effect on booking
  Calendly: If an invitee removes the event from their own calendar,
            the booking in Calendly is NOT cancelled.
  Gap risk: Our webhook is registered on the COACH's target calendar, so
            client-side deletions are invisible to us. Verify this is true
            and document it. No code change needed if confirmed correct.

BEHAVIOR 5 — Google Calendar event marked FREE → ignored for conflict checking
  Calendly: Only events marked BUSY block availability. FREE events
            (common for all-day events added from other sources) are ignored.
  Gap risk: Google's freebusy API already filters to BUSY-only by default.
            Verify our freebusy query does not pass any override that would
            include FREE events. Document the confirmation.

BEHAVIOR 6 — Cancellation made IN OUR SYSTEM → Google Calendar event removed
  Calendly: Cancellations in Calendly are reflected in Google Calendar.
  Gap risk: Our BookingService.cancelBooking() calls Google Calendar API
            to remove/cancel the event. Verify this path is complete and
            handles the case where googleEventId is missing (event was
            manually deleted from GCal before we tried to cancel it —
            Google returns 404, which must be caught gracefully).

BEHAVIOR 7 — Past events have no effect on availability
  Calendly: Past Google Calendar events are irrelevant to slot generation.
  Gap risk: Verify our freebusy query uses timeMin >= now (not some
            earlier anchor date). Verify our reminder cron job skips
            bookings whose startTime is already in the past.

BEHAVIOR 8 — Multiple sub-calendars checked for conflicts
  Calendly: You can select multiple sub-calendars for conflict checking.
            Only BUSY events on those calendars block slots.
  Gap risk: Our freebusy query uses conflictCalendarIds array. Verify
            the array is always populated (default: include at least the
            targetCalendarId itself). Verify the admin UI lets the coach
            select multiple conflict calendars.

--- GAP: BEHAVIOR 2 & 3 — Webhook must detect coach-side changes ---

TASK B.1 — Upgrade the Google Calendar webhook handler to fetch and diff events

  File: src/controllers/booking-webhook.controller.js

  Current behavior: receive notification → flush cache → return 200
  Required behavior: receive notification → fetch changed events → diff →
                     update/cancel bookings as needed → flush cache → return 200

  Implementation:

  When a webhook notification arrives:

  Step 1 — Identify the changed event(s)
    Use the google calendar.events.list() API with:
      calendarId: targetCalendarId for that coach
      updatedMin: (timestamp of last webhook processing — store in a
                   WebhookState collection or in-memory with fallback to now-5min)
      showDeleted: true   ← CRITICAL: needed to detect deletions
      singleEvents: true
      maxResults: 50

  Step 2 — For each changed event returned:
    a. Look up Booking in MongoDB by googleEventId === event.id
    b. If no matching Booking found: skip (not one of ours — e.g. a personal event)
    c. If matching Booking found:

       DELETION / CANCELLATION detection:
         If event.status === 'cancelled'
         OR event is absent (returned with status: 'cancelled' when showDeleted: true)
         AND Booking.status === 'confirmed':
           → Call bookingService.cancelBooking(bookingId, 'coach', 'Cancelled via Google Calendar')
           → This will: update Booking.status, send cancellation email to client
           → Do NOT attempt to delete GCal event again (it's already gone)
           → Log: `Booking ${bookingId} auto-cancelled via GCal webhook`

       RESCHEDULE detection:
         If event.status !== 'cancelled'
         AND (event.start.dateTime !== booking.startTime.toISOString()
              OR event.end.dateTime !== booking.endTime.toISOString()):
           → Call bookingService.rescheduleBooking(bookingId, newStart, newEnd, 'coach_gcal')
           → Log: `Booking ${bookingId} rescheduled via GCal webhook`

  Step 3 — Update stored updatedMin timestamp to now
  Step 4 — Flush availability cache for coachId
  Step 5 — Return 200

  WebhookState collection (or use a simple key-value in OAuthToken):
    coachId: ObjectId
    lastProcessedAt: Date   ← used as updatedMin in events.list query
    lastChannelId: String

  Edge cases to handle:
    - events.list() returns 0 results: nothing changed that affects us, just flush cache
    - Google API call fails: log error, still return 200 (Google will retry)
    - Booking already cancelled: skip silently (idempotent)
    - Multiple changes in one webhook: process all, not just first

TASK B.2 — Add rescheduleBooking method to BookingService

  File: src/services/booking.service.js

  Method: rescheduleBooking(bookingId, newStartTime, newEndTime, triggeredBy)
    triggeredBy: 'coach_gcal' | 'admin' (for audit trail)

  Steps:
    1. Load Booking, verify status === 'confirmed'
    2. Verify newStartTime is in the future (reject past reschedules)
    3. If triggeredBy !== 'coach_gcal':
       Update Google Calendar event with new start/end times
       (if triggeredBy === 'coach_gcal', GCal already has the right time — skip)
    4. Update Booking document:
         startTime: newStartTime
         endTime: newEndTime
         rescheduledAt: new Date()
         rescheduledBy: triggeredBy
         rescheduleHistory: push { from: oldStart, to: newStart, by: triggeredBy, at: now }
    5. Regenerate cancelToken (new JWT, 7-day expiry from new booking time)
    6. Send reschedule confirmation emails (coach + client) — see B.3
    7. Invalidate availability cache for coachId
    8. Return updated booking

  Add to Booking schema:
    rescheduledAt:      Date
    rescheduledBy:      String
    rescheduleHistory:  [{ from: Date, to: Date, by: String, at: Date }]

--- GAP: BEHAVIOR 6 — cancelBooking must handle missing GCal event ---

TASK B.4 — Harden cancelBooking against missing Google Calendar event

  File: src/services/booking.service.js

  In the cancelBooking method, when calling Google Calendar API to
  delete/cancel the event, wrap the API call:

  try {
    await calendar.events.delete({
      calendarId: targetCalendarId,
      eventId: booking.googleEventId,
      sendUpdates: 'none'
    });
  } catch (err) {
    if (err.code === 404) {
      // Event already deleted from GCal (e.g. coach manually deleted it
      // which triggered this cancellation via webhook). Safe to continue.
      logger.info(`GCal event ${booking.googleEventId} already deleted — skipping GCal delete`);
    } else {
      // Unexpected error — still cancel in our DB but log the GCal failure
      logger.error(`Failed to delete GCal event ${booking.googleEventId}:`, err);
    }
  }

  Also handle missing googleEventId:
  if (!booking.googleEventId) {
    logger.warn(`Booking ${bookingId} has no googleEventId — skipping GCal delete`);
  }

  In both cases: proceed to update Booking status and send cancellation email.
  Never let a GCal API failure block the MongoDB update.

--- GAP: BEHAVIOR 7 — Verify past-event safety ---

TASK B.6 — Verify timeMin anchor in freebusy query

  File: src/services/availability.service.js

  Locate where timeMin is set for the freebusy query.

  It must be: Max(fromDate, now + minNoticeHours)
  It must NOT be: fromDate alone (which could be in the past if a cached
                  request is replayed)

  Add guard:
    const safeFrom = DateTime.max(
      DateTime.fromISO(fromDate),
      DateTime.now().plus({ hours: config.minNoticeHours })
    );

  Use safeFrom as timeMin, not raw fromDate.

  Also in the slot generation loop, add an explicit guard per slot:
    // Skip slot if it starts in the past or within minNoticeHours
    if (slotStart <= DateTime.now().plus({ hours: config.minNoticeHours })) {
      continue;
    }

  ## Booking Module — Google Calendar Sync Model

  Our sync model mirrors Calendly's verified behavior:

  | Direction | Trigger | Effect | Implemented |
  |-----------|---------|--------|-------------|
  | GCal → System | Coach deletes/declines event | Booking cancelled, client emailed | ✓ |
  | GCal → System | Coach reschedules event | Booking time updated, client emailed | ✓ |
  | GCal → System | Client deletes event | No effect (we watch coach calendar only) | ✓ |
  | GCal → System | FREE event added | No effect (freebusy API excludes FREE) | ✓ |
  | System → GCal | Booking created | GCal event created in targetCalendarId | ✓ |
  | System → GCal | Booking cancelled | GCal event deleted (graceful on 404) | ✓ |
  | System → GCal | Booking rescheduled by admin | GCal event updated | ✓ |

  Key implementation notes:
  - Webhook handler uses calendar.events.list(showDeleted: true, updatedMin: lastProcessedAt)
    to detect deletions and time changes. It does NOT rely on webhook payload content.
  - targetCalendarId is always included in freebusy conflict check, even if
    conflictCalendarIds array is empty.
  - cancelBooking() is idempotent: handles missing googleEventId and GCal 404 gracefully.
  - rescheduleBooking() triggered by 'coach_gcal' skips GCal update (already done)
    and skips coach email (they know — they made the change).
